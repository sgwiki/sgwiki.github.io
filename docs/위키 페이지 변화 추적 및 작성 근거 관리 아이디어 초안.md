홈페이지(위키)의 서술 근거를 체계적으로 관리하려는 고민은 RAG(Retrieval-Augmented Generation) 시스템을 운영하는 엔지니어들 사이에서 매우 중요한 **'추적성(Traceability)'** 및 **'설명 가능성(Explainability)'** 이슈입니다.

결론부터 말씀드리면, 단순히 Git 커밋 메시지에 의존하는 것은 나중에 특정 문단이 왜 그렇게 작성되었는지 역추적하기 매우 어렵기 때문에, **문서 내에 메타데이터를 직접 관리하는 방식**을 강력히 권장합니다.

---

### 1. 유사한 사례: 어떻게 관리하고 있나?

현재 업계에서 사용하는 방식은 크게 두 가지 흐름이 있습니다.

#### A. Markdown 내 Frontmatter(메타데이터) 활용 (권장)

가장 표준적인 방법은 Markdown 상단이나 각 문단 근처에 구조화된 데이터를 넣는 것입니다. 많은 정적 사이트 생성기(SSG)가 이 방식을 지원합니다.

* **구조 예시:**
```markdown
---
title: AI 모델의 특징
last_updated: 2026-06-19
sources:
  - id: doc_001
    url: "https://arxiv.org/abs/..."
    chunk_hash: "a1b2c3d4..."
---

AI 모델은 확률적으로 생성됩니다. [^source_1]

[^source_1]: 근거: doc_001 (Chunk 5)

```



#### B. Knowledge Graph(지식 그래프) 기반 분리 관리

더 고도화된 시스템에서는 문서와 근거를 1:1로 매핑하는 지식 그래프(Knowledge Graph)를 별도로 구축합니다.

* **방식:** 위키 마크다운 파일에는 `UUID`만 기록하고, 실제 `UUID - Chunk ID` 매핑 테이블은 벡터 DB나 별도의 RDB에 저장합니다.
* **장점:** 여러 문서에서 동일한 소스를 참조할 때 관리가 용이합니다.

---

### 2. 왜 Git 커밋으로 관리하면 안 될까?

Git은 "시간의 흐름(History)"을 기록하는 도구이지, "정보의 논리적 관계(Relationship)"를 기록하는 도구가 아닙니다.

* **역추적의 어려움:** 특정 문단을 보고 "이게 어떤 소스지?"라고 했을 때, Git Blame은 해당 문단을 쓴 '사람'이나 '시점'만 알려줄 뿐, '참조한 데이터셋'을 알려주지 않습니다.
* **데이터 파편화:** RAG 파이프라인에서 문서가 업데이트되거나 청크가 변경될 때마다 Git 이력을 뒤지는 것은 비용이 너무 많이 듭니다.
* **자동화의 한계:** Agent가 자동으로 문서를 생성/수정할 때, Git 커밋 메시지에 소스 ID를 일일이 정교하게 박는 것은 프로그래밍적으로 에러 발생 확률이 높습니다.

### 3. 추천하는 구현 전략: "Embedded Citations"

Agent가 마크다운을 생성할 때, 아래와 같은 구조를 생성하도록 프롬프트를 설계해보세요.

1. **청크 ID 관리:** RAG 파이프라인에서 가져온 각 청크(Chunk)마다 고유한 `Source_ID`를 할당합니다.
2. **마크다운 주석 활용:** 문단 뒤에 보이지 않는 HTML 주석이나 표준 각주(Footnote)를 활용합니다.
* `문단 내용 `


3. **검증 레이어:** 위키 페이지를 저장하기 전, **별도의 검증 Agent**를 두어 마크다운 파일 내의 `Source_ID`가 현재 DB상에 존재하는지 확인(Validation)하게 합니다.

---

### 4. 종합적인 워크플로우 제안

1. **RAG 호출:** 질문에 대해 검색된 청크들에 고유 ID(UUID)를 부여합니다.
2. **생성(Generation):** Agent가 마크다운을 작성하며, 각 답변마다 참조한 `Source_ID`를 문서 하단 또는 각주에 기입합니다.
3. **저장(Storage):** 마크다운 파일은 Git으로 버전 관리하되, **근거 데이터 매핑 정보는 `metadata.json`이나 별도의 Index 파일로 함께 커밋**합니다.
4. **역추적(Traceability):** 사용자가 위키의 특정 문단을 클릭하면, 파일 내의 ID를 읽어 벡터 DB에서 해당 청크를 즉시 불러와 보여줍니다.

이 방식은 향후 위키의 내용이 방대해졌을 때, "이 정보가 최신 데이터로 업데이트되어야 하는지"를 판단하는 **데이터 신선도(Data Freshness) 관리**에도 매우 유리합니다.

지금 만들고 계신 위키의 규모가 어느 정도인가요? 프로젝트의 특성에 맞춰 위 구조 중 더 구체적인 기술 스택(Vector DB 종류 등)을 고민해 볼 수도 있을 것 같습니다.
